02 / Background & Objective

Background & Objective

After Sprint 1 established a quickstart path (QUICKSTART.md, gws error guard, .gitkeep sentinels), two usability gaps remained:

  • GAP-04 (MEDIUM): Demo output paths were not obvious. Generated DOCX lived in deeply nested reports/demo-packages/<slug>/<run-id>/ directories. Scripts printed raw path lines with no visual hierarchy.
  • GAP-06 (MEDIUM): No pre-built example DOCX was committed. A colleague had to install Python and run a script to see what the tool produced.

The objective was to make the demo experience frictionless: a clear banner at script completion, a predictable output/latest-*.docx path, optional auto-open, and committed example files for zero-install preview.

05 / Methodology

Methodology

Sprint 2 followed the plan's 4-phase structure. PHASE-01 and PHASE-02 were implemented in parallel (they had no dependencies on each other), followed sequentially by PHASE-03 and PHASE-04. Each phase was committed, pushed, and reported before beginning the next.

The banner was designed as ASCII-only (no box-drawing characters) to avoid Windows cp1252 encoding issues. The --open flag uses platform detection (os.startfile/open/xdg-open) for cross-platform compatibility.

07 / Findings & Recommendation

Findings & Recommendation

Finding 1: ASCII-only banner was required because Windows cp1252 encoding cannot render Unicode box-drawing characters in the terminal. The initial implementation used ╔═╗ characters and failed on the first test run. Resolution: replaced with +-| ASCII art.

Finding 2: Both Soc Son and Inegol examples were committed (Q-001). The Inegol example is richer (real VCS-3908 project data, 11 VCS v4.4 table types), while the Soc Son example shows the simpler synthetic case. Committing both gives colleagues a choice of what to preview.

Finding 3: By Sprint 2's completion, the colleague-testable demo journey is now fully documented end to end: clone → install → run (Sprint 1) → preview output at predictable paths (Sprint 2).

Recommendation: Proceed to Sprint 3 (Corpus Bundle & Test Cleanup) to provide realistic provenance citations in demo output and clean up stale test skips.
