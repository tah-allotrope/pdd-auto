01 / Executive Summary

Executive Summary

The Soc Son Client Demo Output Upgrade transformed the repository's placeholder-heavy reviewer-facing SOC Son DOCX into a polished client-demo-ready synthetic artifact. Over four phases the work:

  • Traced the root cause of the noisy DOCX to the NoopProvider's intentional [PLACEHOLDER ...] output
  • Defined a separate artifact contract under reports/demo-packages/ reserved for synthetic client-demo output
  • Curated a complete synthetic demo dataset with deterministic assumptions and no review-gated gaps
  • Built a DemoProvider that emits readable synthetic prose instead of placeholders
  • Updated the DOCX export to suppress review telemetry in demo mode while keeping a strong synthetic disclosure
  • Published immutable demo packages with manifest, scorecard, and latest.docx alias
  • Verified end to end: 36 sections, 0 placeholders, 0 low-confidence, 36/36 matched reference, average grounding 0.7

The result is a one-command workflow (python scripts/run_demo.py) that produces a client-safe synthetic PDD sample under reports/demo-packages/ with zero review-gate noise and aligned quantification numbers.

02 / Background & Objective

Background & Objective

The repository builds Verra VCS Project Design Documents (PDDs) for waste-to-energy carbon credit projects. The existing Soc Son review workflow produced DOCX output that was faithful to its design — it used the NoopProvider, which intentionally emits [PLACEHOLDER ...] section bodies for human-in-the-loop review. When the latest Soc Son sample was opened in Word, the document was dominated by placeholder markers, review-required warnings, and a reviewer-issues appendix — making it unusable as a client-facing sample.

The objective was to produce a separate, client-demo-ready Soc Son sample that is readable, internally consistent, and explicitly synthetic, without weakening the existing reviewer-facing artifact contract.

04 / Assumptions & Constraints

Assumptions & Constraints

  • ASM-001: The noisy DOCX was a workflow-contract mismatch, not a DOCX export defect — confirmed by inspecting the run JSON
  • ASM-002: The fastest path to a client-demo artifact is curated synthetic content, not wiring a real API-backed text model
  • ASM-003: The demo config is a better base than the spreadsheet mapper because it avoids the mapper's intentionally blocked review items
  • CON-001: Synthetic demo content must remain explicitly labeled as demo content
  • CON-002: The reviewer-facing workflow must keep its current behavior unchanged
  • CON-003: PDF export is optional; DOCX is the guaranteed artifact surface
  • DEC-001: The client-demo path builds on Phase 05 rather than overloading the Vietnam review package path
05 / Methodology

Methodology

The work followed a 4-phase strategy:

  1. Phase 1 — Contract Definition: Trace the root cause in code and docs, define acceptance rules for demo artifacts (no [PLACEHOLDER markers, no REVIEW REQUIRED bullets, aligned numbers, strong synthetic disclosure), and reserve reports/demo-packages/ as the canonical publication path.
  2. Phase 2 — Curated Synthetic Inputs: Extend create_demo_project_input() to write a companion assumptions YAML with demo_curated provenance. Populate all required fields with internally consistent synthetic values. Introduce a distinct provenance type so demo-safe facts are not conflated with review-gated spreadsheet assumptions.
  3. Phase 3 — Demo Narrative & Clean Export: Add a deterministic DemoProvider producing readable section prose. Update SectionOrchestrator to suppress reviewer-noise escalation for demo runs. Modify DOCX export to use a strong disclosure, rename the appendix to "Assumption Summary", and omit the reviewer-issues appendix. Add publish_demo_package() and wire it into the CLI.
  4. Phase 4 — Verification & Fresh Artifact: Add regression tests for zero-placeholder output, quantification consistency, and end-to-end publication. Run the demo workflow, verify the DOCX programmatically, update documentation, and publish the final artifact.

06 / Phase-by-Phase Analysis

Phase-by-Phase Analysis

Phase 1 — Root-Cause Trace & Artifact Contract

Confirmed the noisy Soc Son DOCX is expected behavior from the current workflow: NoopProvider.draft_section() intentionally emits [PLACEHOLDER ...] bodies and REVIEW REQUIRED issues. The demo benchmark also accepted placeholder-heavy output with 36 placeholder sections and 36 low-confidence sections. Defined the client-demo contract: no placeholders in the body, aligned cross-section numbers, strong cover disclosure, compact appendix. Selected reports/demo-packages/<project-slug>/<run-id>/ as the canonical path with summary-only telemetry.

Phase 2 — Curated Synthetic Demo Inputs

Extended create_demo_project_input() to write demo_socson_like.assumptions.yaml alongside the existing demo config. The assumptions register uses demo_curated provenance for all entries and carries blocked_review_paths: [] (no review gates). All quantitative fields are internally consistent: baseline, project, leakage, net, and crediting period total. Tests confirm the config and assumptions stay in sync.

Phase 3 — Demo Narrative & Clean Export

Implemented a deterministic DemoProvider with curated section text for key sections (1.1, 1.10, 3.4, 3.5, 4.1, 4.2, 4.4, 5.2) and a fallback template for the rest. Updated SectionOrchestrator so demo runs save provider="demo" and suppress review-noise escalation. Modified DOCX export: the disclaimer reads "synthetic client-demo sample", reviewer notes are suppressed in the body, the assumption appendix is renamed to "Appendix A - Assumption Summary", and the reviewer-issues appendix is omitted entirely for demo runs. Published publish_demo_package() and wired it into both run_demo_benchmark() and scripts/run_demo.py.

Phase 4 — Verification & Fresh Client Artifact

Added 3 new regression tests validating all sections have zero placeholders, quantification arithmetic is consistent (net = baseline - project - leakage, total = net * years), and the end-to-end demo package publishes correctly. Ran a fresh demo workflow producing 36 HIGH-confidence sections with 0 placeholders, 36/36 matched reference sections, and an average grounding score of 0.7. Verified the DOCX XML programmatically: no [PLACEHOLDER markers, strong disclosure present, correct appendix, key quantification numbers (98,000 / 21,000 / 2,000 / 75,000 / 750,000) all present. Updated README and generated this final report.

09 / Risks / Open Questions

Risks / Open Questions

  • Deterministic prose may feel generic. The DemoProvider maps section keys to hard-coded template text. A live demo may expose that the prose is formulaic. Mitigation: keep demos focused on workflow capability, not narrative flair.
  • Demo artifacts could be mistaken for real project evidence. Mitigated by strong cover-page disclosure, package naming including demo, and separate reports/demo-packages/ vs reports/review-packages/ paths.
  • Operators may confuse review packages and demo packages. Mitigated by separate directories, separate CLI output labels, and explicit README language.
  • Single-project validation. Only the Soc Son-like fixture is tested. A second WTE project would strengthen confidence.
