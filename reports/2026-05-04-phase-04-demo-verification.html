06 / Limitations / Second-Best Alternative

Limitations / Second-Best Alternative

  • Demo prose is deterministic, not generative. The DemoProvider maps section keys to hard-coded template text, so the output is the same every run. This is acceptable for client demos but not for real PDD drafting.
  • PDF conversion skipped. No local DOCX-to-PDF converter was available. DOCX remains the guaranteed artifact surface per CON-003.
  • Single-project validation. The Soc Son-like fixture is the only validated demo dataset. A second WTE project would strengthen confidence in the demo workflow.
  • Manual inspection limited to automated checks. DOCX XML verification confirmed no placeholders, proper appendix, and correct numbers, but a human should still open the file before a real client demo.

Second-Best Alternative

Instead of a separate DemoProvider, we could have wired the NoopProvider to emit shorter placeholder text and restyled the export to hide the review gates. That approach was rejected because the run JSON itself would remain placeholder-heavy, making the DOCX fragile to re-export and harder to debug.

08 / Open Questions / Next Phase Seeds

Open Questions / Next Phase Seeds

  • Should the demo workflow wire a real LLM next? The demo provider is deterministic; the next useful step could be replacing it with OpenAIProvider or OllamaProvider for generative output.
  • Should demo packages get a PDF variant? The plan marks PDF as optional (CON-003), but a client-ready PDF is a common request. Worth adding when a stable conversion pipeline exists.
  • Should the demo path include Drive upload? The review path already supports --upload-review-docx. A --upload-demo-docx flag could be added for demos that need a shareable Drive link.
