01 / Executive Summary

Executive Summary

What was built

The repository now spans corpus ingestion, schema and rule encoding, retrieval-backed drafting, review workflow, export/upload surfaces, and a reproducible Phase 05 benchmark flow with markdown scorecards and a one-command demo runner.

What matters

The architecture proves that a low-cost, local-first workflow is viable. The benchmark also shows that with `NoopProvider` the system still produces placeholders, so the next leverage point is provider quality rather than more scaffolding.

02 / Background & Objective

Background & Objective

The project goal was to build an agentic, near-zero-cost assistant for waste-to-energy carbon credit PDD drafting using Verra-style source documents, local storage and indexing, and optional LLM generation only where needed. The plan deliberately avoided a monolithic hallucination-prone document generator and instead focused on section-level provenance, review gates, and measurable workflow quality.

Across five phases, the repo moved from empty scaffolding to a working CLI that can ingest a WTE corpus, normalize and parse it, build an FTS5 retrieval index, draft every canonical section, apply review and consistency checks, export the result, and benchmark the workflow against a Soc Son-like reference case.

04 / Assumptions & Constraints

Assumptions & Constraints

Assumptions

  • A narrow Verra-style WTE corpus is sufficient to prove the workflow before expanding methodologies.
  • Section-level provenance and reviewability matter more than one-shot document fluency at this stage.
  • The Soc Son sample is representative enough to benchmark structure and comparison logic.

Constraints

  • Keep recurring infra cost near zero using local files, SQLite FTS5, and `gws` instead of managed services.
  • Human review remains mandatory for high-risk methodology and quantitative sections.
  • Current benchmark still uses `NoopProvider`; DOCX export also depends on `python-docx` being installed in the local runtime.
05 / Methodology

Methodology

The work followed the plan’s phased architecture: first establish a clean local corpus and bucket, then encode canonical section structure and methodology rules, then build retrieval-backed drafting, then add explicit review/export surfaces, and finally benchmark the resulting workflow on a realistic demo case. Verification emphasized tests, deterministic artifacts, and a real smoke path rather than narrative claims.

The final benchmark used a Soc Son-like `ProjectInput`, ran the existing drafting and review pipeline, parsed the normalized Soc Son reference, and generated comparison artifacts summarizing match coverage, provenance presence, placeholder density, and a simple grounding score. That made the Phase 05 recommendation observable rather than subjective.

06 / Phase-by-Phase Analysis

Phase-by-Phase Analysis

PHASE-01

Established the CLI foundation, Drive inventory, download/normalization pipeline, and a first WTE corpus bucket. This created the source-of-truth corpus and readiness reporting needed for every later phase.

PHASE-02

Converted observed Verra/WTE structure into a canonical five-section schema, a validated `ProjectInput` contract, and methodology rule wrappers. This kept later drafting grounded in explicit structure rather than prompts alone.

PHASE-03

Added local SQLite FTS5 retrieval and a section orchestrator that assembles prompts, examples, and project facts. The provider abstraction kept the system local-first while leaving room for real models later.

PHASE-04

Added rule-based review checks, cross-section consistency checks, state persistence, DOCX export, and Drive upload wrappers. This turned draft text into a reviewable handoff package.

PHASE-05

Completed the missing benchmark layer with reproducible demo input, scorecards, comparison notes, CLI support, and smoke-run verification. The benchmark showed complete workflow execution but also confirmed that placeholder output still dominates until a real provider is wired.

What changed across phases

The project evolved from corpus plumbing to increasingly opinionated structure, then to execution and finally to measurement. That sequence mattered: the benchmark only has meaning because ingestion, schema, retrieval, and review were already explicit and testable.

07 / Findings & Recommendation

Findings & Recommendation

Finding 1: The planned architecture is now implemented end to end. The repo can create a demo config, execute the draft-and-review path, and emit benchmark artifacts without ad hoc manual file wrangling.

Finding 2: The benchmark proves structural workflow integrity, not useful draft quality. The most recent smoke run drafted 36 sections, matched all 36 against the reference structure, and achieved an average grounding score of 0.501, but every section remained `UNSUPPORTED` because the current provider is still `NoopProvider`.

Finding 3: The most valuable next investment is provider quality, not additional orchestration. More scaffolding would not change the core benchmark outcome while placeholder text dominates.

Recommendation: Harden the same bucket by wiring one real provider and rerunning the exact Phase 05 benchmark before expanding to a second WTE bucket. If the grounded first-draft quality remains weak even after that, pause and revisit architecture. If it improves materially, then expand to a second project for broader validation.

08 / Implementation Path

Implementation Path

Step 1

Install the missing local runtime dependency for export (`python-docx`) so the benchmark always emits DOCX evidence alongside markdown artifacts.

Step 2

Wire one real provider into the existing provider registry, keeping the benchmark command unchanged so results stay comparable.

Step 3

Rerun `pdd-agent benchmark --input configs/projects/demo_socson_like.yaml` and compare grounding, low-confidence count, and reviewer burden against the current placeholder baseline.

Step 4

If the rerun is materially better, create a second demo project from the same methodology family before expanding buckets or making external quality claims.

09 / Risks / Open Questions

Risks / Open Questions

Risks

  • The current benchmark can overstate structural success because every section still carries provenance even when the text is only a placeholder.
  • A single Soc Son-like case is vulnerable to overfitting and should not be treated as broad evidence.
  • Provider integration may surface new cost, latency, or grounding issues that the placeholder path does not reveal.

Open questions

  • Should the first real-provider benchmark use local Ollama for cost discipline or OpenAI for a cleaner quality signal?
  • What threshold should define a “materially useful first draft” for high-risk sections like baseline and additionality?
  • Does the project need stronger section-level semantic scoring once true generation is enabled?
